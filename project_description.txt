COBOT MAGIC VLA FINE-TUNING PROJECT HANDOFF
============================================

1. Project overview
-------------------

This repository contains fine-tuning, evaluation, and real-robot inference
integrations for Vision-Language-Action (VLA) and robot learning models on the
Cobot Magic bimanual robot.

The three primary model families are:

* OpenVLA-OFT, based on OpenVLA 7B.
* SmolVLA, using the LeRobot training and policy stack.
* NVIDIA GR00T N1.7.

The project goal is not only to obtain a low training loss. The goal is to
adapt each model to the same Cobot Magic dataset and compare real-robot
behavior under controlled conditions. Important comparison dimensions include:

* task success rate;
* visual grounding and correct object selection;
* correct left/right arm selection;
* grasp acquisition and gripper timing;
* placement accuracy;
* motion smoothness and stability;
* collision and unsafe-target frequency;
* out-of-distribution behavior;
* reproducibility across checkpoints and machines.

The real robot communicates with GPU inference servers through a shared ZeroMQ
JSON protocol. Each server must convert its model-specific output into absolute
14D joint targets before replying to the robot client.


2. Repository structure
-----------------------

openvla-oft/
    OpenVLA-OFT fine-tuning and inference code. The Cobot Magic integration
    includes a direct LeRobot-like dataset loader, 3-camera input, proprio
    conditioning, FiLM, LoRA, continuous L1 or diffusion action heads, and a
    ZeroMQ server.

lerobot/
    LeRobot source tree and the SmolVLA integration. Cobot Magic additions
    include joint-only dataset handling, relative joint actions, optional
    left-arm loss weighting, training commands, and a ZeroMQ server.

Isaac-GR00T/
    NVIDIA Isaac-GR00T source tree with the Cobot Magic modality config,
    fine-tuning entry points, weighted action loss support, offline evaluation,
    and a ZeroMQ server.

checkpoints/
    Deployment convention for exported checkpoints. This directory may exist
    outside the repository, for example:

        /home/dual4090/workdir/apanasevich/checkpoints

    Large checkpoints must not be committed to Git. Some development machines
    may temporarily contain imported checkpoint directories in the repository
    root; these must remain ignored. The authoritative checkpoint location for
    each experiment requires verification from its train/evaluation record.

logs/
    There is no single mandatory repository-level log directory. Each pipeline
    currently has its own convention:

    * openvla-oft/logs/ and openvla-oft/runs/
    * lerobot/logs_smolvla/
    * Isaac-GR00T/logs/ and legacy Isaac-GR00T/logs_groot/

    New commands should follow the current README for the corresponding model.

Dataset root
    The dataset is external to the repository and is normally passed through:

        export DATASET_DIR=/absolute/path/to/cobot_magic_sber

    Never assume that the dataset is available on an inference-only server.
    Dataset statistics required for inference must travel with the checkpoint.


3. Dataset
----------

The project uses the local dataset named cobot_magic_sber. It follows a
LeRobot-like layout with metadata, parquet episode data, and videos.

Core properties:

* Robot: Cobot Magic, bimanual setup.
* Number of cameras: 3.
* Recorded frame rate: 15 FPS.
* camera_0: right wrist camera.
* camera_1: left wrist camera.
* camera_2: high/front/central camera.

The dataset metadata may expose 26-dimensional state and action vectors:

* the first 14 values are bimanual joint values;
* the remaining values contain FK-derived end-effector information.

The current control system and the current model integrations use joint-space
control. The normal Cobot Magic representation is therefore:

* proprio dimension: 14;
* action dimension: 14;
* dimensions 0 through 6: left arm, including its gripper dimension;
* dimensions 7 through 13: right arm, including its gripper dimension;
* expected gripper indices in the current code: 6 and 13;
* action chunk length: 24.

The raw dataset feature order and the 14D slicing must be verified whenever a
new dataset version is introduced. Do not silently combine the 14D joints with
the FK end-effector coordinates.


4. Global reproducibility rules
-------------------------------

Use seed 42 for all experiments unless a documented experiment explicitly
requires another seed. Some upstream tools have different defaults, so seed 42
must be passed explicitly where supported.

Every experiment record must preserve:

* random seed;
* train/validation episode split and split seed;
* exact checkpoint path;
* model version and Git commit;
* dataset version or immutable dataset path;
* action representation: absolute or relative;
* normalization mode and saved statistics;
* camera count and camera mapping;
* action and proprio dimensions;
* action chunk length;
* complete training command;
* complete inference command;
* GPU type and number of GPUs;
* important environment/package versions.

Do not compare two models if one of these fields is unknown. Mark missing
information as "requires verification" rather than inferring it from a folder
name.


5. Common Cobot Magic constants
-------------------------------

OpenVLA-OFT requires these environment-level constants:

    ROBOT_PLATFORM=COBOT_MAGIC
    COBOT_MAGIC_NUM_ACTIONS_CHUNK=24

The shared semantic constants are:

    ACTION_DIM=14
    PROPRIO_DIM=14
    NUM_ACTIONS_CHUNK=24

Camera mapping:

    camera_0 -> right wrist
    camera_1 -> left wrist
    camera_2 -> high/front/central

Joint order:

    0..6  -> left arm
    7..13 -> right arm

The action mode at inference must match the action mode used during training.
This is a checkpoint property, not a global model-family property. Legacy and
new checkpoints from the same model family may use different modes.


6. Action representation
------------------------

Absolute action:

    a_t = q_(t+1)

The predicted action is already the next absolute joint target. The server must
return it directly. Adding the current joint position again is incorrect.

Relative or delta action:

    a_t = q_(t+1) - q_t

At inference, after denormalization:

    absolute_target = current_qpos + predicted_delta

Normalization must be computed for the representation actually used by the
model. For relative-action training, compute the delta first and then compute
or apply relative-action normalization statistics. Do not normalize absolute
targets and subtract the state afterward.

A train/inference mismatch is safety critical:

* treating an absolute target as a delta adds q_t twice and can make the robot
  jump or rotate violently;
* treating a delta as an absolute target can send joints toward values near
  zero or otherwise invalid configurations;
* using statistics from the wrong representation can produce plausible-looking
  but numerically unsafe outputs.

Inspect dataset_statistics.json or the model-specific checkpoint config before
running a checkpoint. Relative action statistics should generally differ
substantially from absolute proprio statistics and are often centered near
zero. This is a diagnostic, not a substitute for an experiment record.


7. OpenVLA-OFT
--------------

Location:

    openvla-oft/

Purpose:

    Fine-tune OpenVLA 7B for Cobot Magic with 3 images, 14D proprio, FiLM
    conditioning, LoRA, and a continuous action head.

Important training parameters include:

    --vla_path openvla/openvla-7b
    --dataset_format lerobot
    --dataset_name cobot_magic_sber
    --use_l1_regression True
    --use_diffusion False
    --use_film True
    --num_images_in_input 3
    --use_proprio True
    --lora_rank 32
    --action_head_hidden_dim 2048
    --action_head_num_blocks 1

The action head hidden dimension and block count are experiment parameters.
Configurations such as hidden dimension 4096 or two blocks are valid ablations,
but are not guaranteed improvements.

OpenVLA-OFT supports two continuous action approaches:

L1 regression head
    Directly regresses a continuous action chunk. It is simpler and faster at
    inference, but a single deterministic trajectory may be limiting for
    multimodal bimanual manipulation.

Diffusion action head
    Models an action distribution through iterative denoising. It may better
    represent multiple plausible bimanual trajectories, but requires more
    training and slower inference. This remains a hypothesis to test on the
    Cobot Magic evaluation protocol.

Critical checkpoint components must come from the same training step:

* lora_adapter/adapter_model.safetensors;
* action_head--<step>_checkpoint.pt;
* proprio_projector--<step>_checkpoint.pt;
* vision_backbone--<step>_checkpoint.pt when FiLM is enabled;
* dataset_statistics.json;
* processor/config files.

Important LoRA/FiLM constraint:

* The trained LoRA adapter must actually be loaded with
  PeftModel.from_pretrained or correctly merged into the base model.
* Creating get_peft_model(...) at inference without loading the adapter creates
  newly initialized LoRA matrices and invalidates the checkpoint.
* Do not apply a LoRA adapter again on top of already merged LoRA weights.
* In the current FiLM inference path, the clean base model is loaded from
  --base_model_path, the trained lora_adapter is loaded, and then the saved
  FiLM vision backbone is restored.
* LoRA rank must match lora_adapter/adapter_config.json.

Common OpenVLA-OFT failure modes:

* LIBERO constants loaded because ROBOT_PLATFORM was not set.
* ACTION_DIM or PROPRIO_DIM differs from 14.
* action head hidden dimension does not match the saved state dict.
* FiLM enabled for a non-FiLM checkpoint or disabled for a FiLM checkpoint.
* random or double-applied LoRA.
* action head, projector, vision backbone, and adapter from different steps.
* incorrect unnorm_key.
* absolute checkpoint launched with relative conversion, or the reverse.


8. SmolVLA / LeRobot
--------------------

Location:

    lerobot/

SmolVLA uses the LeRobot training pipeline. Cobot Magic training normally uses:

* joint-only 14D state and action;
* relative joint action targets when
  dataset.relative_joint_actions=true;
* chunk_size=24;
* n_action_steps=24 in the current training config;
* 3 renamed image inputs;
* a local pretrained SmolVLA base checkpoint;
* optional loss weighting for the left arm.

At inference, the ZeroMQ server loads a checkpoint or its pretrained_model/
subdirectory. For relative checkpoints, it postprocesses the predicted delta
and then adds the current 14D proprio exactly once before returning absolute
targets.

max_actions controls how many actions from the predicted chunk are returned to
the client. It does not change the trained chunk size. Predicting 24 actions and
returning or executing the first N actions is receding-horizon control, not a
shape mismatch. Smaller N replans more frequently; larger N is more open-loop.

The default enabled image augmentation set is:

* brightness: (0.8, 1.2);
* contrast: (0.8, 1.2);
* saturation: (0.5, 1.5);
* hue: (-0.05, 0.05);
* sharpness: (0.5, 1.5);
* affine rotation: (-5 degrees, 5 degrees);
* affine translation: (0.05, 0.05).

With max_num_transforms=3, up to three sampled transforms are applied to a
frame. The training commands also enable random transform order. These image
augmentations do not modify state or action tensors.


9. GR00T N1.7
-------------

Location:

    Isaac-GR00T/

The project fine-tunes NVIDIA GR00T N1.7 for Cobot Magic through a custom
modality configuration and the NEW_EMBODIMENT/new_embodiment tag.

The current Cobot Magic modality config uses:

* cam_high, cam_left_wrist, and cam_right_wrist;
* state.all_arms with 14 dimensions;
* action.all_arms with 14 dimensions;
* a 24-step action horizon;
* relative joint action representation.

GR00T uses a diffusion/flow-matching style action policy. In the current
project, it has appeared more stable than the tested OpenVLA-OFT L1 checkpoint
on the real robot. This is an empirical observation from current tests, not a
general model ranking.

The project supports action-loss reweighting:

* gripper indices: 6 and 13;
* example gripper loss weight: 4.0;
* example late-chunk start: timestep 10;
* example late-chunk loss weight: 2.0.

The default upstream-compatible weights are 1.0. Values 4.0 and 2.0 are
experiment settings in the current README, not universal constants.

This weighting may help with grasp closure and release timing, and the
diffusion-style policy may help with multimodal pick-and-place trajectories.
These are plausible explanations for observed behavior and require controlled
ablation before being treated as conclusions.

GR00T-specific requirements:

* load examples/CobotMagic/cobot_magic_config.py;
* use the embodiment tag stored by the checkpoint;
* keep the checkpoint statistics and processor files;
* do not add proprio in the ZMQ server if GR00T's processor has already
  converted relative actions to absolute targets.


10. Inference architecture
--------------------------

The model server runs on a GPU machine and binds a ZeroMQ REP socket, normally:

    tcp://0.0.0.0:5055

The robot client uses a ZeroMQ REQ socket and sends one JSON request containing:

* type="inference";
* images_camera_0: base64 JPEG list for the right wrist;
* images_camera_1: base64 JPEG list for the left wrist;
* images_camera_2: base64 JPEG list for the high/front camera;
* instruction: natural-language task string;
* proprio: current joint sequence, normally shape [1, 14];
* optional metadata.

The server:

1. decodes the three JPEG streams;
2. maps cameras to model-specific keys;
3. validates and normalizes proprio;
4. runs policy inference;
5. denormalizes model actions;
6. converts relative actions to absolute targets when required;
7. validates shape and safety constraints;
8. returns:

       {"actions": [[...14 values...], ...]}

The client executes the first N actions and then requests another chunk.

Before moving the physical robot, verify:

* camera order;
* joint order;
* action and proprio dimensions;
* checkpoint action mode;
* normalization key and statistics;
* returned action shape;
* finite values;
* maximum target delta or another safety clamp;
* that only one process owns the selected ZMQ port.


11. Debugging checklist
-----------------------

When behavior is unstable, check these items before blaming model quality:

[ ] ROBOT_PLATFORM=COBOT_MAGIC is set for OpenVLA-OFT.
[ ] COBOT_MAGIC_NUM_ACTIONS_CHUNK=24 is set where required.
[ ] ACTION_DIM=14 and PROPRIO_DIM=14 are printed at startup.
[ ] camera_0 is right wrist.
[ ] camera_1 is left wrist.
[ ] camera_2 is high/front.
[ ] Joint dimensions are left 0..6 and right 7..13.
[ ] The checkpoint uses the expected absolute or relative action mode.
[ ] Relative actions are denormalized before adding current_qpos.
[ ] Absolute actions are not added to current_qpos.
[ ] The correct dataset statistics and unnorm_key are loaded.
[ ] Current proprio lies inside or near the training distribution.
[ ] The LoRA adapter is loaded, not randomly initialized.
[ ] A merged LoRA is not being applied a second time.
[ ] LoRA rank matches adapter_config.json.
[ ] Action head hidden dimension and block count match training.
[ ] FiLM mode and number of cameras match training.
[ ] All component files come from the same checkpoint step.
[ ] A merged checkpoint includes valid config and model safetensors files.
[ ] The client and server agree on action shape [N, 14].
[ ] The client is not applying delta conversion after the server returns
    absolute targets.
[ ] max_actions and client execution count produce an intentional replanning
    interval.
[ ] Unsafe target jumps are rejected before reaching the robot.
[ ] Wrong-arm behavior is logged separately from object localization failure.


12. Evaluation protocol
-----------------------

Models must be evaluated under matched conditions:

* same dataset version;
* same train/validation split;
* same camera mapping and image preprocessing;
* same action representation where the experiment is intended to compare
  architectures rather than representations;
* same objects and tasks;
* same object placement distribution;
* same robot start pose;
* same instruction wording;
* same physical environment and lighting;
* same action execution frequency and replanning policy;
* same number of trials.

For each trial, log:

* success or failure;
* selected object;
* selected arm;
* object localization error;
* grasp success and number of attempts;
* gripper close/open timing;
* placement success;
* trajectory smoothness;
* collisions or near-collisions;
* safety-filter rejection;
* OOD warnings;
* checkpoint and full inference command.

Report per-task results and aggregate results. A single successful video is not
enough to compare models.


13. Current known conclusions and hypotheses
--------------------------------------------

Current observations:

* GR00T currently appears more stable on Cobot Magic real-robot tests.
* Tested OpenVLA-OFT L1 checkpoints have shown instability, object localization
  errors, and occasional wrong-arm selection.
* Checkpoint assembly and normalization errors can mimic poor model quality.
  These must be excluded before interpreting behavior.

Working hypotheses:

* Deterministic L1 regression may be less suitable than diffusion-style action
  modeling for multimodal bimanual manipulation.
* A larger action head may help capacity, but it is an ablation and not a
  guaranteed fix.
* Relative joint actions may improve stability compared with absolute
  next-state targets.
* Gripper and late-chunk loss weighting may improve grasp/release timing.

Requires verification:

* Whether OpenVLA-OFT diffusion consistently outperforms its L1 head on this
  dataset.
* Whether a larger OpenVLA action head improves grounding or only increases
  optimization cost.
* Whether observed GR00T advantages remain under a fully matched evaluation
  protocol.
* The best number of client-executed actions per 24-step model chunk.
* The best loss weights for each arm and gripper without overfitting.

Before changing model architecture, verify action representation,
normalization, checkpoint component consistency, and real-robot input
distribution.

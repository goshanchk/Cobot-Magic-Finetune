Core terms
----------

Episode:

    One complete demonstration/trajectory of a task. Example: pick up an
    object and place it into the plastic container.

Timestep:

    One time step inside an episode. Since fps=15, consecutive timesteps are
    about 1/15 sec = 0.0666667 sec apart.

Trajectory:

    The full sequence inside an episode: images, robot states, actions,
    timestamps, and language instruction.

Rollout:

    A run of a trained policy in an environment or on a robot. The dataset
    contains demonstrations/trajectories, not online rollouts.

Chunk:

    A technical storage grouping. Here chunks usually contain 1000 episodes.
    A chunk is not a task and not an episode.

Observation:

    What the model sees: images, robot state/proprioception, and language.

Action:

    What the model is trained to predict. In this dataset action has special
    semantics: action[t] is the next patched robot state, state[t+1].

    action[t] = patched observation.state[t+1]

This means action is an absolute next-state target, not a delta action and not
low-level torque/motor command.

meta/stats.json
~~~~~~~~~~~~~~~

Contains global normalization stats for observation.state and action:

    mean, std, min, max, q01, q99

Critical issue:

    stats.json contains only 14 values for observation.state and action.
    But info.json and parquet files contain 26-dimensional state/action.

Likely explanation:

    The original dataset had 14 arm/joint dimensions. Later the dataset was
    patched with 12 EEF pose dimensions using forward kinematics. Per-episode
    stats were updated, but global stats.json appears stale.

Training risk:

    Do not blindly use stats.json to normalize 26-dim state/action. It will
    either fail by shape mismatch or silently normalize only part of the action
    depending on the loader.

How a training sample is constructed
------------------------------------

For one timestep t:

    Inputs:
        image from camera_0 at frame_index=t
        image from camera_1 at frame_index=t
        image from camera_2 at frame_index=t
        observation.state[t]
        language instruction from task_index

    Target:
        action[t]

In this dataset:

    action[t] = observation.state[t+1]

For episode 0:

    task_index = 0
    instruction = "Pick up the roll of tape and place it in the plastic container."

The model does not usually learn by predicting the next image and fitting an
action to that image. In behavior cloning/imitation learning it directly learns:

    policy(images, state, instruction) -> expert_action

Here expert_action is the next 26-dimensional robot state.

VLA learning intuition
----------------------

VLA means Vision-Language-Action.

Typical components:

    images -> vision encoder / vision transformer -> visual tokens
    text -> language model / text transformer -> language tokens
    robot state -> projection MLP -> state/proprio tokens
    fused tokens -> transformer/policy head -> action vector

The output can be a continuous vector:

    action = [a0, a1, ..., a25]

The training loss compares this predicted vector to the dataset action:

    loss = L1/MSE(predicted_action, expert_action)

The model learns how to change each joint because it repeatedly sees examples
of similar scenes, instructions, current states, and expert next actions.

There can be many possible valid joint configurations. This is called action
ambiguity. Behavior cloning does not solve all possible plans; it learns the
style and local next-step choices present in the demonstrations.

Why current state matters:

    Images show objects and scene layout.
    Language says what to do.
    State tells the model where the robot already is.
    The next action is only a short step at 15 Hz, not a full task plan.

Known risks and issues
----------------------

1. stats.json is stale/incomplete

    info.json and parquet say state/action are 26-dimensional.
    stats.json has only 14 values for state/action.

    Risk:
        Wrong normalization during training.

    Recommended action:
        Recompute global stats for all 26 dims or configure loader to use
        correct 26-dim stats from episodes_stats.jsonl/parquet.

2. action semantics may not match model defaults

    This dataset stores:
        action[t] = state[t+1]

    Some VLA pipelines expect:
        delta action
        EEF-only action
        joint-only action
        action chunks
        discretized action tokens

    Recommended action:
        Before training OpenVLA-OFT, GreenVLA, or GR00T, inspect each model's
        expected action format and convert if needed.

3. Very short episodes exist

    length_min = 2 timesteps.

    Risk:
        Broken/incomplete demonstrations can harm training or metrics.

    Recommended action:
        List and inspect episodes with length below a threshold, e.g. < 20 or
        < 50 timesteps.

4. No explicit success labels

    Risk:
        Failed demonstrations may be included unless filtered elsewhere.

    Recommended action:
        Use viewer/DataQA/filter artifacts and sample visual inspection.

5. Validation split is not in info.json splits

    validation_episodes.json has 30 episodes, but info.json says all episodes
    are train.

    Risk:
        Training code may train on validation episodes unless explicitly
        configured.

6. Language has typos/inconsistencies

    Examples:
        "balck marker"
        "place it in" vs "place it into"
        missing punctuation

    Risk:
        Usually acceptable for robust VLA models, but can matter for task
        grouping, evaluation, and instruction normalization.

7. AV1 video codec

    Videos are AV1.

    Risk:
        Some loaders/environments may lack AV1 decode support or decode slowly.

    Recommended action:
        Test the exact training dataloader on a few samples before long runs.

8. Gripper semantics are unclear

    There is no explicit gripper field. q6 for each arm may be gripper-like,
    but this must be confirmed.

9. EEF poses are FK-derived

    EEF xyz/rpy were patched using forward kinematics and a URDF.

    Risk:
        If URDF/sign conventions are wrong, EEF fields may be systematically
        off. Joint scale signs are listed in info.json and should be respected.

Current compatibility intuition
-------------------------------

OpenVLA-OFT:

    Dataset is close to LeRobot v2, but may not work directly if OpenVLA-OFT
    expects a different action representation. Need to inspect the exact local
    OpenVLA-OFT training config. Likely needs at least action normalization fix
    and possibly action conversion.

GreenVLA:

    Main research target. It may be suitable if GreenVLA supports multi-camera
    image input, language instructions, proprioception, and 26-dim continuous
    action or a configurable action head. Need exact GreenVLA data adapter. The
    14-vs-26 stats issue must be handled.

GR00T N1.7:

    Likely requires a converter or schema adapter, because GR00T-style datasets
    commonly expect a specific embodiment/action schema and modality naming.
    Need to inspect the local GR00T data format requirements.

Dataset suitability for research
--------------------------------

11.28 hours is a meaningful small/medium fine-tuning dataset for a focused
pick-and-place-like domain, especially if starting from a pretrained VLA.

Expected behavior:

    From scratch training: not enough.
    Fine-tuning pretrained VLA: plausible.
    High risk of overfitting if tasks/objects/scenes are repetitive.

Likely batch size on 2x RTX 4090:

    Depends heavily on model size, image count, action horizon, precision, and
    gradient checkpointing. For large VLA fine-tuning, expect small per-GPU
    batches and gradient accumulation. A realistic starting point is often
    per-GPU batch 1-4 for heavy models, larger only with frozen encoders/LoRA or
    shorter action chunks.

Most important blockers before any long training run:

    1. Fix/avoid stale 14-dim stats.json.
    2. Confirm model expected action semantics.
    3. Confirm gripper meaning.
    4. Confirm AV1 dataloader support.
    5. Filter or inspect suspicious short episodes.
    6. Make validation split explicit in training config.
